# Automatic Sharpness Adjustment When Reducing Digital Images

## Подробное описание алгоритма, который вырисовывается на текущий момент

> Этот документ — инженерная реконструкция алгоритма на основе доступных публикаций и их описаний.
> Он **не претендует на paper-exact reproduction**, если конкретная деталь не подтверждена напрямую.
>
> Цель документа — собрать в одном месте:
> - что можно считать подтверждённым,
> - какой pipeline логически следует из доступных источников,
> - что можно реализовывать уже сейчас,
> - какие части пока остаются гипотезой.

## 1. Источники, на которых основана реконструкция

Основная связка работ:

1. **Algorithm and Program to Downsizing the Digital Images** (2016)
2. **Saturation Adjustment in the Digital Images Processing** (2016)
3. **Automatic Sharpness Adjustment when Reducing Digital Images** (2018)

На текущий момент именно эта тройка лучше всего задаёт общий облик алгоритма:

- работа 2016 года про **downsizing** даёт базовый pipeline уменьшения изображения;
- работа 2018 года про **automatic sharpness adjustment** добавляет механизм автоматического выбора коэффициента резкости;
- работа про **saturation adjustment** помогает понять цветовую философию автора и вероятную дисциплину работы с gamut/clipping.

## 2. Что можно считать подтверждённым

### 2.1. Уменьшение изображения выполняется в линейном цветовом пространстве

Из статьи про downsizing следует, что:

- преобразования должны выполняться в **линейном цветовом пространстве**;
- промежуточные вычисления должны использовать **вещественные числа**;
- scaling должен выполняться отдельно как самостоятельный этап;
- после масштабирования применяются **controlled sharpness gain** и **contrast leveling**.

### 2.2. Резкость усиливается после downscale, а не вместо него

По доступным описаниям sharpening — это **post-resize stage**, а не часть декодирования и не предварительная обработка исходника.

Иначе говоря, общий каркас таков:

```text
input -> linearize -> downscale -> postprocess(sharpness, contrast) -> output
```

### 2.3. Автоматический выбор коэффициента резкости основан на ограничении артефактов

Из статьи 2018 года известно, что проблема фиксированного sharpening coefficient заключается в том, что разные изображения по-разному реагируют на усиление резкости.

Ограничивающим фактором служат артефакты:

- светлые halo,
- тёмные halo,
- визуальное утолщение тонких линий,
- очень светлые и очень тёмные точки,
- искажения цветов после clipping.

### 2.4. Артефакты связываются с выходом цветовых значений за gamut

Одна из главных идей статьи 2018 года:

- sharpening делает тёмные значения темнее и светлые светлее;
- в результате часть цветов может выйти за допустимый охват / диапазон;
- после этого включается clipping;
- clipping приводит к искажению тона и слиянию различных цветов;
- именно это используется как измеримый индикатор артефактов.

### 2.5. Для выбора коэффициента используется зависимость P(Sharpness)

В статье 2018 года явно фигурирует зависимость:

```text
P = P(Sharpness)
```

где `P` — доля цветовых значений, вышедших за допустимый gamut / range.

### 2.6. Зависимость P(Sharpness) аппроксимируется полиномом третьей степени

Это один из наиболее конкретных известных фактов.

Иными словами, алгоритм не просто перебирает sharpening “на глаз”, а строит приближение вида:

```text
P_hat(s) = a*s^3 + b*s^2 + c*s + d
```

### 2.7. Итоговый коэффициент выбирается по допустимому порогу P0

В доступных материалах присутствует пример использования порога:

```text
P0 = 0.1%
```

Далее для разных изображений выбирается своё значение `Sharpness`, при котором фактическое `P` близко к целевому `P0`.

Это практически однозначно указывает на критерий:

> выбрать максимальное допустимое усиление резкости, которое ещё не превышает бюджет артефактов.

## 3. Общая идея алгоритма

На текущий момент наиболее правдоподобная реконструкция выглядит так:

1. Изображение декодируется.
2. Выполняется перевод в линейное цветовое пространство.
3. Изображение уменьшается в линейном пространстве.
4. На уменьшенном изображении запускается postprocess.
5. Для нескольких значений коэффициента sharpening измеряется уровень артефактов `P(s)`.
6. По полученным точкам строится кубическая аппроксимация `P_hat(s)`.
7. По заданному `P0` находится допустимое значение `s*`.
8. С найденным `s*` выполняется финальная коррекция резкости.
9. Результат ограничивается допустимым диапазоном.
10. Выполняется обратное преобразование в выходное цветовое пространство.
11. Результат сохраняется.

В кратком виде:

```text
input
-> decode
-> linearize
-> downscale
-> probe sharpening strengths
-> compute P(s)
-> fit cubic P_hat(s)
-> solve P_hat(s) = P0
-> apply final sharpening(s*)
-> clip / map to valid output range
-> output
```

## 4. Подробный pipeline

## 4.1. Вход и декодирование

Предполагается, что вход — обычное цифровое изображение в одном из стандартных форматов (PNG, JPEG и т.д.), закодированное в стандартном нелинейном RGB-представлении.

Практическая инженерная трактовка:

```text
u8 / u16 RGB -> normalized RGB [0..1]
```

На этом шаге пока **не делается sharpening** и **не делается resize**.

---

## 4.2. Перевод в linear RGB

Это один из самых надёжно подтверждённых этапов.

Все ключевые вычисления, связанные с уменьшением изображения, должны выполняться уже после перевода из нелинейного RGB в линейное пространство.

Практически:

```text
srgb -> linear rgb
```

### Почему это важно

Если уменьшать изображение прямо в gamma-encoded пространстве, то смешивание цветов становится физически некорректным. Для downscale это особенно заметно, потому что там постоянно происходит усреднение и интерполяция.

### Следствие для реализации

Внутреннее представление изображения в библиотеке должно быть float-based:

```text
LinearRgbImage<f32>
```

или аналогичный тип.

---

## 4.3. Downscale в линейном пространстве

После linearization выполняется уменьшение изображения.

### Что известно

Известно, что:

- resize — это отдельный этап;
- он выполняется в линейном пространстве;
- он имеет “естественную физическую интерпретацию”.

### Что неизвестно

Пока нет точной подтверждённой формулы resize kernel / resampling strategy.

Поэтому в инженерной реализации допустимо использовать:

- area/box-style downscale,
- или иной качественный, понятный и воспроизводимый метод,

но при этом честно документировать, что это **практическая замена**, а не доказанное совпадение с оригинальным кодом автора.

---

## 4.4. Post-resize stage

После уменьшения изображения запускается postprocess.

Подтверждённо известно, что в него входят:

- **sharpness gain**,
- **contrast leveling**.

Это означает, что у автора уже был pipeline, где после resize применялась некая форма:

```text
downscaled_image -> sharpen -> contrast leveling
```

или

```text
downscaled_image -> contrast leveling -> sharpen
```

Порядок этих двух шагов в автоматическом режиме пока точно не подтверждён.

### Практический вывод

В архитектуре библиотеки sharpening и contrast leveling нужно держать **раздельными модулями**, чтобы позже можно было:

- переставлять их местами,
- включать/выключать contrast leveling,
- уточнить авторский порядок без переделки всего pipeline.

---

## 4.5. Параметризуемый sharpening

Ключевая переменная статьи 2018 — это коэффициент усиления резкости, условно `Sharpness` или `s`.

То есть нужен sharpening-оператор с непрерывным или квазинепрерывным параметром `amount`.

На уровне инженерной реконструкции это очень похоже на семейство алгоритмов вида:

- unsharp mask,
- high-pass enhancement,
- edge-emphasis filter.

### Почему это выглядит правдоподобно

Потому что:

- используется именно коэффициент усиления,
- строится зависимость `P(Sharpness)`,
- в примерах встречаются значения вида `1.09`, `1.81`, `2.17`.

Это больше похоже на коэффициент силы эффекта, чем на параметры сложной модели.

### Что пока неизвестно

Точная формула sharpening filter по доступным материалам не подтверждена.

Поэтому для первой реализации допустимо взять:

- прозрачный и хорошо документированный unsharp mask,
- с явным параметром `amount`,
- а позже заменить реализацию, если удастся достать точную формулу.

---

## 4.6. Определение метрики артефактов P(s)

Это центральная идея всей схемы.

Статья 2018 года подсказывает, что артефакты удобно характеризовать через долю цветовых значений, выходящих за допустимый gamut после sharpening.

### Практическая трактовка для первой реализации

Пусть после sharpening мы имеем изображение в linear RGB float.

Тогда можно определить:

```text
P(s) = число компонент, для которых c < 0 или c > 1
       --------------------------------------------
              общее число цветовых компонент
```

То есть проверяется каждый канал каждого пикселя.

Например:

- RGB-изображение 100×100 содержит `100 * 100 * 3 = 30000` компонент;
- если 30 компонент вышли за диапазон, то:

```text
P = 30 / 30000 = 0.001 = 0.1%
```

### Почему это хорошее стартовое определение

- оно очень близко по смыслу к формулировке статьи;
- оно детерминированное;
- его легко тестировать;
- оно хорошо подходит под автоматическую оптимизацию;
- его можно позже заменить на более авторски-точную версию.

### Возможные будущие уточнения

Позже можно будет заменить метрику на:

- долю пикселей, а не каналов;
- оценку в другом цветовом пространстве;
- оценку в ортогональной системе цветовых координат автора;
- более сложный artifact score с halo penalty.

Но для первой работающей версии текущая трактовка достаточно сильная.

---

## 4.7. Сэмплирование функции P(s)

Чтобы выбрать итоговую силу sharpening, нужно не одно значение `P`, а несколько измерений для разных коэффициентов `s`.

То есть берётся набор пробных значений:

```text
S = {s1, s2, s3, ..., sn}
```

Для каждого выполняется:

```text
downscaled -> sharpen(amount = s_i) -> compute P(s_i)
```

В результате получается набор точек:

```text
(s1, P1), (s2, P2), ..., (sn, Pn)
```

### Практический смысл

Это уже полноценная эмпирическая функция “насколько сильно sharpening начинает ломать допустимый диапазон”.

---

## 4.8. Кубическая аппроксимация P(s)

Следующий подтверждённый шаг — кубическая аппроксимация.

По измеренным точкам строится приближение:

```text
P_hat(s) = a*s^3 + b*s^2 + c*s + d
```

### Зачем это нужно

Чтобы:

- не полагаться на грубый перебор;
- получить гладкую оценку между sampled points;
- решать обратную задачу по заданному порогу `P0`.

### Инженерное замечание

В реальной реализации нужно предусмотреть fallback:

- если кубическая аппроксимация плохо обусловлена,
- если данные слишком шумные,
- если полином даёт странные корни,

то алгоритм не должен ломаться.

Практичный fallback:

- прямой поиск по sampled points,
- линейная/кусочная интерполяция,
- выбор наибольшего допустимого значения из реально измеренных.

---

## 4.9. Решение уравнения P_hat(s) = P0

После построения аппроксимации задаётся допустимый уровень артефактов `P0`.

Далее решается уравнение:

```text
P_hat(s*) = P0
```

И выбирается итоговый коэффициент `s*`.

### Наиболее правдоподобный критерий выбора

Если корней несколько, нужно выбирать:

> наибольшее допустимое значение sharpening, которое ещё не превышает artifact budget.

Именно этот критерий лучше всего согласуется с общей постановкой задачи и примером из статьи.

---

## 4.10. Финальное применение найденного sharpening

После нахождения `s*` выполняется отдельный финальный проход sharpening:

```text
final = sharpen(downscaled, s*)
```

Это важный момент.

Probe phase и final apply phase должны быть разделены:

- сначала оценка и подбор,
- потом уже итоговое применение.

Такой дизайн:

- проще тестировать,
- проще профилировать,
- проще объяснять в GUI,
- проще возвращать диагностику пользователю.

---

## 4.11. Ограничение диапазона и обратное преобразование

После финального sharpening результат может содержать значения вне допустимого диапазона рабочего представления.

Поэтому на выходе логично выполнить:

1. clip / remap в допустимый диапазон;
2. преобразование из linear RGB обратно в выходное цветовое пространство;
3. кодирование в файл.

Практически:

```text
linear float -> clamp -> gamma encode -> u8/u16 output
```

---

## 5. Итоговый псевдокод алгоритма

```text
input: image I, target size W x H, artifact threshold P0

1. Decode input image
2. Convert image from nonlinear RGB to linear RGB
3. Downscale image in linear space to target size
4. Optionally apply contrast leveling baseline stage
5. Select probe sharpening strengths S = {s1, ..., sn}
6. For each s in S:
   6.1 Apply sharpening with strength s
   6.2 Measure P(s) = proportion of out-of-range color values
7. Fit cubic approximation P_hat(s)
8. Solve P_hat(s) = P0
9. Select valid solution s*
10. Apply final sharpening using s*
11. Clamp / map image into valid output range
12. Convert image from linear RGB back to output color space
13. Encode and save output image
14. Return diagnostics
```

## 6. Что можно считать самым важным выводом для реализации

### 6.1. Алгоритм управляется не “красотой”, а ограничением артефактов

Это очень важная инженерная особенность.

Во многих практических системах sharpening подбирается либо вручную, либо через эвристику “сделать резче”.

Здесь же идея гораздо более строгая:

- резкость увеличивается,
- но только до тех пор, пока контролируется доля артефактных цветов.

То есть это **constrained optimization**, а не субъективный auto-enhance.

### 6.2. Resize и sharpening должны быть раздельными этапами

Это один из ключевых архитектурных выводов.

Нельзя проектировать библиотеку как “магическую функцию resize_with_everything_inside”.

Нужны отдельные модули:

- color conversion,
- resize,
- sharpening,
- artifact metric,
- fitting,
- solver,
- pipeline orchestration.

### 6.3. Линейное пространство — не опция, а фундамент

Если хочется сделать реализацию, похожую по духу на статьи, нельзя ограничиться наивной обработкой `u8 RGB` в sRGB.

### 6.4. Метрика P удобна как для CLI, так и для GUI

Поскольку `P` — понятная численная величина, её удобно:

- выводить в CLI,
- показывать в Tauri GUI,
- использовать в диагностике,
- использовать в regression tests.

## 7. Что можно реализовать уже сейчас

Уже сейчас можно написать рабочую первую версию библиотеки, даже без полного доступа к точной формуле из статьи.

### Рабочий MVP может выглядеть так

1. Загрузка изображения.
2. `sRGB -> linear RGB`.
3. Качественный downscale.
4. Параметризуемый unsharp mask.
5. Метрика:

```text
P = доля значений вне диапазона [0,1]
```

6. Набор пробных `s`.
7. Измерение `P(s)`.
8. Кубическая аппроксимация.
9. Поиск `s*` по `P0`.
10. Финальное применение.
11. Возврат результата и диагностики.

Это уже будет сильная, честная, инженерно обоснованная версия алгоритма.

## 8. Что остаётся неизвестным

Ниже список мест, где пока нельзя уверенно заявить точную paper-level формулу.

### 8.1. Точный алгоритм downscale

Пока не подтверждена конкретная формула resize kernel / integration strategy.

### 8.2. Точный sharpening operator

Неизвестно:

- exact formula,
- kernel shape,
- radius,
- наличие edge-aware logic,
- точный смысл коэффициента `Sharpness`.

### 8.3. Точное определение P

Не до конца ясно:

- считается ли `P` по каналам,
- по пикселям,
- или в специальной цветовой системе автора.

### 8.4. Роль contrast leveling

Известно, что contrast leveling существует в pipeline, но пока не ясно:

- используется ли он всегда,
- в каком месте точно стоит,
- влияет ли на вычисление `P(s)`.

### 8.5. Численная схема fitting / solving

Неизвестны детали:

- сколько точек использует автор,
- как именно строится cubic fit,
- как выбирается корень при нескольких решениях,
- есть ли дополнительные ограничения на диапазон `Sharpness`.

## 9. Наиболее разумная инженерная интерпретация на сегодня

На текущий момент самый сильный и честный вариант трактовки такой:

### Подтверждённая основа

- линейное цветовое пространство,
- float intermediate calculations,
- downscale как отдельный этап,
- sharpening после уменьшения,
- артефакты через выход за gamut,
- кубическая аппроксимация `P(Sharpness)`,
- выбор `Sharpness` по порогу `P0`.

### Практические замены для неизвестных частей

- качественный downscale вместо неустановленного авторского ядра,
- unsharp mask как baseline sharpening,
- `P` как доля каналов вне `[0,1]`,
- fallback на sampled search, если cubic fit неустойчив.

### Архитектурный вывод

Библиотеку нужно проектировать не как один монолит, а как набор независимых модулей:

- `color`
- `resize`
- `contrast`
- `sharpen`
- `metrics`
- `fit`
- `solve`
- `pipeline`

Именно такой дизайн даст возможность:

- быстро собрать рабочий MVP,
- позже приблизиться к paper-exact version,
- без боли добавить Tauri GUI,
- отдельно тестировать каждый этап.

## 10. Вывод

На основании доступных источников уже вырисовывается достаточно чёткий алгоритм:

- изображение декодируется,
- переводится в линейное цветовое пространство,
- уменьшается,
- на уменьшенном результате для ряда коэффициентов резкости измеряется доля выходящих за диапазон цветовых значений,
- эта зависимость аппроксимируется полиномом третьей степени,
- по заданному лимиту артефактов выбирается итоговое значение `Sharpness`,
- после чего применяется финальная коррекция резкости и формируется выходное изображение.

Это описание уже достаточно конкретно, чтобы начинать реализацию на Rust как честный engineering-first prototype.

## 11. Краткая формулировка в одном абзаце

Алгоритм, который вырисовывается на данный момент, представляет собой downscale pipeline в линейном цветовом пространстве с последующей автоматической настройкой sharpening: для нескольких значений коэффициента усиления резкости на уменьшенном изображении измеряется доля цветовых компонент, выходящих за допустимый диапазон, затем эта зависимость аппроксимируется кубическим полиномом, после чего по заранее выбранному порогу допустимых артефактов определяется итоговый коэффициент резкости, который и применяется к финальному изображению.
